## ⚠️ Coverage Gaps / Contradictions Flagged By This Revision — Read Before Implementation

None of these block sign-off, but each requires an explicit implementation decision that is **not fully spelled out** by the ratified resolutions. Test cases below assert the recommended interpretation; if implementation diverges, the corresponding test must be updated **with the divergence recorded in the migration/function comment**, not silently.

1. **GAP-1 — `populated_at` filtering is asymmetric between rolls and completions, and this is easy to implement wrong.** The resolution text says the flag "counts only rolls/completions after `populated_at`" and cites `d100_roll_log` entries with `rolled_at >= populated_at` as the mechanism. But there is **no completion-log table** — `complete_d100()` only maintains a single running `times_completed` counter with no timestamped history to filter. Analysis: this is actually a **non-issue by construction**, not a true gap — `complete_d100()`'s existing guard (`WHERE task_name IS NOT NULL`) means a completion can only ever be recorded after a slot has task content, which (per DQ-1/DQ-2) only exists after population. Therefore `times_completed` is inherently "completions since population" already; no additional timestamp filtering is needed on the completions side. **Only the roll-count side needs `populated_at`-based filtering** (via `d100_roll_log`), because pre-population populate-me rolls DO touch `last_rolled`/`times_rolled` per DQ-1(B). **This must be an explicit, commented design choice in the flagging query** — an implementer who filters both sides symmetrically (or neither) will either double-penalize or under-count. See TC-FLAG-06 through TC-FLAG-06c, TC-POPAT-04.

2. **GAP-2 — Rolls-since-population count must come from `d100_roll_log`, not `times_rolled`, and the flagging query's `>=10` threshold column changes meaning.** The original `>=10` threshold (TC-FLAG-01..04, unchanged) was written against lifetime `times_rolled`. Per DQ-2's resolution, the corrected query must compute "rolls since population" as `(SELECT count(*) FROM d100_roll_log WHERE roll = m.roll AND rolled_at >= m.populated_at)`, and gate the `>=10` threshold off **that** count, not `m.times_rolled` directly, for any slot that has ever had a pre-population roll. For slots populated at creation time (no pre-population rolls ever occurred), the two counts are identical, so this only bites already-migrated slots that were rolled as empty before this feature shipped. **Backfill semantics (below) determine how much this matters in practice.**

3. **GAP-3 — Backfill semantics for `populated_at` on already-populated slots is undecided and materially affects the flag for all 56 existing populated slots.** Neither resolution comment states what `populated_at` should be set to for the 56 slots that are already populated today (they were never "empty then populated" — they've had content since or before this feature existed). Two candidate backfills, both defensible:
   - **(a) Backfill `populated_at = created_at`** (or a fixed migration timestamp, e.g. `now()` at migration time) for all currently-populated slots — treats "always had content" as "populated since the beginning," so ALL of their historical rolls count (no rolls-since-population filtering kicks in for legacy slots; behaves exactly like the current, unfiltered `times_rolled` math).
   - **(b) Leave `populated_at = NULL`** for currently-populated slots — but then the filtering query (`rolled_at >= populated_at`) breaks (`NULL` comparisons never satisfy `>=`), which would make **every legacy slot's rolls-since-population count come back 0**, incorrectly suppressing the flag for slots the issue's own motivating example (roll 96, 9/19=47.4%) depends on catching.
   Recommend **(a)** as the only backfill that preserves TC-FLAG-05 (the live motivating regression case) and doesn't silently defang the flag for the entire pre-existing 56-slot population. Test TC-POPAT-02/03 assert (a); flag prominently for Coder — **this is the single highest-risk implementation detail in the whole migration**, since getting it wrong doesn't error, it just silently stops the flag from ever firing on legacy data.

4. **GAP-4 (minor, non-blocking) — DQ-6's "oldest `last_rolled`" re-admission ordering is still marked "flagged for I)ruid confirmation at morning review" in the resolution comment itself, i.e., the orchestrator's own resolution is provisional.** Test cases proceed against the stated interpretation (stateless, oldest-`last_rolled`-first) per the resolution, but sign-off on TC-ANTI-06/TC-ANTI-07 test **results** should be held pending that morning-review confirmation, in case it flips. This does not block writing/committing the test cases themselves.
