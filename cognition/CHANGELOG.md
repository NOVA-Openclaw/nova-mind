## Unreleased

### Fixed (#506 ‚Äî branch-safety check for schema-sync commits/pushes)

- **`sync_schema_to_github()` now asserts branch == `main` before any dump, commit, or push** ([#506](https://github.com/NOVA-Openclaw/nova-mind/issues/506)) ‚Äî Prior to this fix, the function never verified the checked-out branch of its working clone (`NOVA_MIND_DIR`, normally `~/.openclaw/workspace/nova-mind`). When the clone was left on a stale feature branch after unrelated work concluded (the root cause observed in production from 2026-07-17 16:23 UTC onward), every subsequent sync attempt committed schema dumps onto that branch and then failed to push (branch diverged from its own remote) ‚Äî a masked failure: sync events logged `False` to the `events` table, but nothing alerted, so `database/schema.sql` on `main` silently drifted from the live database for days. A new `_ensure_on_main(command, table_name)` check now runs as the first statement inside the existing git-lock critical section (`try` block, immediately after `fcntl.flock(..., LOCK_EX | LOCK_NB)` succeeds) and before the `pgschema dump` step ‚Äî on **every call**, with no process-lifetime cache, since the listener is a long-running daemon and branch state can change between NOTIFY events (e.g. ops intervention, a stray manual commit).
- **Remediation matrix:**
  - **Already on `main`:** `git fetch origin`, then `git merge --ff-only origin/main`. If origin was ahead, the clone fast-forwards before the dump/commit ‚Äî the resulting commit is correctly parented on the fetched tip, not the stale local tip. No alert is sent for this routine catch-up case.
  - **Wrong branch or detached `HEAD`:** `git checkout main`, then the same fetch + fast-forward as above. The previously-checked-out branch/commit is left byte-identical before and after ‚Äî no commit is ever created on it, either before or during the checkout. This is the exact issue #506 root-cause scenario.
  - **`main` diverged from `origin/main`** (the `--ff-only` merge fails): abort loudly. No commit is created, nothing is pushed, and any pre-existing local-only commits are left completely untouched (no auto-discard, no `reset --hard`, no rebase attempted ‚Äî reconciliation is deferred to a human). Returns `(False, None)`.
  - **Checkout or fetch fails outright** (dirty working tree blocking `git checkout main`, unreachable remote, etc.): abort loudly the same way ‚Äî `(False, None)`. Uncommitted edits to tracked files are never discarded; git itself refuses the checkout/merge when a dirty file would be clobbered, and `_ensure_on_main` treats that nonzero exit as an abort signal rather than forcing through with `--force` or `stash`.
  - Every abort path returns `(False, None)` ‚Äî the function never returns `(True, ...)` when the underlying branch problem was not actually resolved, and no unguarded exception can escape `_ensure_on_main` (all git calls are checked via `.returncode`, not `check=True`).
- **New alert path: `_send_branch_alert(found_branch, command, table_name, reason, stderr=None)`** (#506) ‚Äî Mirrors `_send_push_alert()`'s structure (own `try/except Exception` that logs and never re-raises, so an `agent_chat` outage cannot turn a correctly-detected failure into a false success). Sends `send_agent_message('schema-sync', ..., ['nova'])` with the expected branch (`main`), the branch actually found, and reason-specific manual-recovery commands (`diverged` ‚Üí `git rebase origin/main && git push origin main`; `fetch failed` ‚Üí check remote connectivity then `git merge --ff-only origin/main`; `checkout failed` ‚Üí `git stash && git checkout main && ...`). Truncated git stderr (first 500 chars) is included when available.
- **Lock discipline unchanged** (#506) ‚Äî `_ensure_on_main()` sits inside the same `try` block whose `finally` clause unconditionally releases the git lock (`fcntl.flock(..., LOCK_UN)` + fd close) on every exit path, including the new early-return-on-abort paths introduced by this fix. This was necessary for the race invariant to hold structurally, not just by luck: two listener instances racing the same `schema_changed` NOTIFY (production has observed up to 5 concurrent instances ‚Äî see #271/#461) still serialize on the file lock, and whichever instance acquires it second re-reads branch state fresh rather than trusting a stale in-memory value from the first.
- **No changes to existing push retry/backoff/failure-classification logic** (#506) ‚Äî `_classify_push_failure()`, `_send_push_alert()`, `MAX_PUSH_ATTEMPTS`, and `PUSH_BACKOFF_DELAYS` are byte-for-byte unchanged; they only run after `_ensure_on_main()` returns `True`.

Under #506, that exact scenario is now correctly reclassified: the branch-safety check fast-forwards the clone *before* the commit is made, so the push succeeds instead of failing. The assertions were updated to match (alert-call count reduced accordingly); this is the intended, in-scope consequence of the fix, not a weakened regression guard ‚Äî the true-divergence abort case (local commit + different origin commit) is covered separately and unchanged (Test 18). #### Known follow-ups (not part of this fix)
- **#507** ‚Äî No automated test currently drives `_send_push_alert()`'s `non-fast-forward` classification branch, since the only #399-era scenario that used to reach it ("clone behind origin") is now handled by `_ensure_on_main()`'s fast-forward before a commit is ever attempted. The code path is untouched and structurally correct by inspection; it's now only reachable via a narrow same-call TOCTOU race (something pushes to `origin/main` in the gap between `_ensure_on_main()`'s fast-forward and the function's own `git push` a few lines later, both inside the same held lock). Recommend a follow-up test using a fake-git `push` hook to simulate that race.
- **#508** ‚Äî `pg-notify-listener.py`'s `agent_chat` alerts (`_send_push_alert()`, pre-existing, and the new `_send_branch_alert()`) both hardcode sender `'schema-sync'`. Live staging integration testing (SE Run #474 Step 6b) discovered `send_agent_message()` now enforces `LOWER(p_sender) = session_user`, and no `schema-sync` database role exists ‚Äî so every listener alert, in every environment, has silently failed to deliver since approximately 2026-07-12 (last successful `schema-sync` `agent_chat` message: 2026-07-12 16:04 UTC). Not a #506 regression ‚Äî the sender was already hardcoded in #399-era code, and this fix's error handling degrades correctly: the alert-send failure is logged (never raised), and `_ensure_on_main()` still returns `False` regardless of whether the alert itself was delivered. Recommended fix (separate issue/PR): send as the connecting DB user (`_agent_chat_env['PGUSER']` / `getpass.getuser()`) with a `[schema-sync]` message prefix, or add a sanctioned service-sender mechanism to `send_agent_message()`.

### Fixed (#399 ‚Äî direct git push for schema-sync, retry/backoff, failure alerting)

- **`sync_schema_to_github()` now pushes schema commits directly to `origin` instead of delegating to the ephemeral subagent `gidget` via `agent_chat`** ([#399](https://github.com/NOVA-Openclaw/nova-mind/issues/399)) ‚Äî Gidget is a spawn-time-only subagent with no persistent listener, so prior `send_agent_message('schema-sync', ..., ['gidget'])` work orders were never drained (158 dead messages, 25 unpushed commits accumulated between 2026-04-09 and 2026-07-05 before this fix). `sync_schema_to_github()` now runs `git push origin main` itself, inside the same git lock and `try/finally` that already serializes and cleans up the local commit step.
- **Failure-class-differentiated retry policy** (#399) ‚Äî Git push stderr is classified via `_classify_push_failure()` into `transient`, `auth`, or `non-fast-forward`. Transient failures (network blips, timeouts, unreachable remote) retry up to `MAX_PUSH_ATTEMPTS = 3` with exponential backoff (`PUSH_BACKOFF_DELAYS = [2, 4]` seconds between attempts). Auth failures and non-fast-forward rejections fail fast (no retry) since retrying cannot fix either condition. Each attempt has a `PUSH_TIMEOUT = 60`-second ceiling via `subprocess.run(..., timeout=PUSH_TIMEOUT)`.
- **Non-fast-forward is alert-only, no auto-rebase** (#399) ‚Äî If `origin/main` has diverged, the listener does not attempt an automatic `fetch`/`rebase`. It fails fast and alerts with explicit manual-reconciliation steps (`git fetch origin && git rebase origin/main && git push origin main`). Automatic rebase was judged too risky for an unattended background service to perform on schema commits.
- **Push failure alert now goes to `nova` instead of `gidget`** (#399) ‚Äî `_send_push_alert()` sends a `send_agent_message('schema-sync', ..., ['nova'])` on permanent push failure (retries exhausted, or fail-fast on auth/non-fast-forward). `nova` is a live listener, unlike `gidget`. The alert message includes the failure class, commit hash, truncated git stderr (first 500 chars), and failure-class-specific manual recovery instructions. The alert-send path is wrapped in its own `try/except` and never propagates exceptions ‚Äî a failure to reach `agent_chat` is logged, not raised, so it can't crash the NOTIFY main loop.
- **`sync_schema_to_github()` return contract on permanent failure** (#399) ‚Äî Returns `(False, commit_hash)` (previously `(False, None)`) so callers retain the commit hash of the un-pushed local commit for logging/diagnostics even when the push itself failed.
- **Push subprocess sets `OPENCLAW_AGENT_ID=gidget`** (#399) ‚Äî The global git pre-push hook (see `cognition/docs/system-level-controls.md`) only allows full push access when `OPENCLAW_AGENT_ID` is `gidget` or `git-agent`. Since this push now runs mechanically from the schema-sync script rather than from an actual Gidget-spawned process, the subprocess environment sets `OPENCLAW_AGENT_ID=gidget` so the hook allows it through. **Known follow-up (not part of this fix):** this reuses Gidget's identity for a non-Gidget process; a dedicated `schema-sync` identity with its own pre-push hook allowlist entry is flagged as a follow-up rather than implemented here. The `agent_chat` sender identity is unaffected and still reads `schema-sync`.
- **Stale `handle_schema_change()` manual-recovery guidance updated** (#399) ‚Äî The fallback instructions surfaced on total sync failure (`pgschema dump` failure, or after push exhausts retries) previously ended with step 4, "Delegate push to Gidget via agent_chat" ‚Äî no longer accurate since delegation was removed. Now reads `git push origin main` (a step the on-call agent/human can run directly).
- **`notify_clawdbot` dead code annotated** (#399) ‚Äî Added a comment clarifying that `notify_clawdbot` is a documented no-op (dead webhook, `nova-workspace#4`) and that the operative failure/success notification path for schema-sync is `send_agent_message` via `_agent_chat_env`, not the Clawdbot webhook call still present at the end of `handle_schema_change()`.

#### Tests (#399)
- `cognition/tests/test_pg_notify_listener_issue_399.py` ‚Äî 16 new tests covering: happy-path push, no-op path (clean status.porcelain), transient-then-success retry, permanent transient failure (retries exhausted ‚Üí alert), auth failure (fail-fast, no retry, alert), non-fast-forward failure (fail-fast, no retry, alert with rebase instructions), per-attempt timeout handling, backoff delay timing, alert payload contents (commit hash, failure class, stderr truncation, recipient `['nova']`, sender `'schema-sync'`), lock acquisition/release across all success and failure paths (including mid-push exception), and confirmation that no `agent_chat` message addressed to `gidget` is ever sent from the schema-sync path (regression gate for the issue's acceptance criterion). See `cognition/tests/TEST-CASES-ISSUE-399.md` for the full test design this suite implements.

#### Issues Closed
- #399 ‚Äî schema-sync delegates git push to non-listening subagent 'gidget' via agent_chat (158 dead work orders, 25 unpushed commits)

### Changed (#244 ‚Äî per-gateway agent scoping + UPPERCASE bootstrap source labels)

- **`agent-config-sync` query switched to `get_agent_export_rows()`** ([#244](https://github.com/NOVA-Openclaw/nova-mind/issues/244)) ‚Äî The inline `AGENTS_QUERY` in `cognition/focus/agent-config-sync/src/sync.ts` previously used `WHERE instance_type != 'peer' AND model IS NOT NULL`, which returned the same rows regardless of which peer gateway was connecting. It now calls the DB function `get_agent_export_rows()` (lives in `database/schema.sql`, owned by newhart) which uses `session_user` to scope results: the connecting peer's own row (emitted as `is_default = TRUE`) plus subagents linked to that peer via `parent_agents` array overlap. Each peer gateway therefore receives only its own agent set in `agents.json`, fixing the default-agent confusion on Newhart/Graybeard gateways where they were previously seeing NOVA's subagents and falling back to NOVA as default. Supersedes the prior `instance_type != 'peer'` filter described below.
- **`get_agent_bootstrap()` source literals capitalized** ([#244](https://github.com/NOVA-Openclaw/nova-mind/issues/244)) ‚Äî The five tier labels emitted by `get_agent_bootstrap()` in `database/schema.sql` are now UPPERCASE: `UNIVERSAL`, `GLOBAL`, `DOMAIN:<name(s)>`, `WORKFLOW:<name>`, `AGENT` (previously lowercase). The bootstrap-context hook handler composes paths as `` `db:${row.source}/${row.filename}` ``, so callers now see `db:UNIVERSAL/CORE.md`, `db:DOMAIN:Project Leadership/ORCHESTRATION.md`, `db:AGENT/SOUL.md`, etc. The casing aligns the source field with the `agent_bootstrap_context.context_type` column (UNIVERSAL/GLOBAL/DOMAIN/AGENT/SYSTEM/WORKFLOW) and makes path tiers visually distinct from the `db:` namespace prefix.
- **Coordinated with nova-openclaw [#243](https://github.com/NOVA-Openclaw/nova-openclaw/issues/243)** ‚Äî The new UPPERCASE `db:TIER/...` synthetic identifiers are preserved verbatim by the gateway's bootstrap-files sanitizer (`sanitizeBootstrapFiles()` in nova-openclaw), so they never get corrupted into workspace-rooted filesystem paths. Both PRs must ship together ‚Äî see staging deploy notes on PR #244.
- **`notify_agent_config_changed()` trigger fires on any row change** ‚Äî Was already broadened in an earlier change; documenting here for completeness because the docs for the trigger now also reflect the wider net.

- **Removed system defaults from `agents.json` output** ‚Äî `agents.defaults.subagents` (e.g. `maxSpawnDepth`) is no longer written by the sync plugin; it stays in `openclaw.json`. The `agent_system_config` table still exists but its values are applied at install time, not at runtime via sync. ([#174](https://github.com/nova-openclaw/nova-cognition/issues/174), [#169](https://github.com/nova-openclaw/nova-cognition/issues/169))
- **Agent filter changed to `instance_type != 'peer'`** ‚Äî Previously the query used `IN ('primary', 'subagent')`; now it excludes only peer agents, so any new `instance_type` values are automatically included without requiring a query change. ([#174](https://github.com/nova-openclaw/nova-cognition/issues/174), [#170](https://github.com/nova-openclaw/nova-cognition/issues/170))
- **NOTIFY trigger simplified to unconditional** ‚Äî The `notify_agent_config_changed()` trigger previously fired only on changes to specific columns (model, fallback_models, etc.). It now fires on any row change, so newly added columns (like `is_default`) are automatically captured without trigger updates. ([#174](https://github.com/nova-openclaw/nova-cognition/issues/174), [#170](https://github.com/nova-openclaw/nova-cognition/issues/170))

### Added (continued from #163)
- **System-wide subagent config synced from `agent_system_config` table** ‚Äî The `agent-config-sync` plugin now also reads the `agent_system_config` table and syncs system-wide defaults (e.g. `maxSpawnDepth`) into `agents.json` under `agents.defaults.subagents`. Changes to the table automatically propagate via the existing `agent_config_changed` LISTEN/NOTIFY channel. A new DB trigger (`notify_system_config_changed`) and migration (`163-system-config-trigger.sql`) are applied by the installer. Supported keys: `max_spawn_depth` (integer, clamped to 1‚Äì5 ‚Üí `maxSpawnDepth`); `max_concurrent_subagents` (reserved for future use). - **`agent_system_config` table and trigger in installer** ‚Äî `agent-install.sh` now ensures the `agent_system_config` table exists, applies the system config trigger migration, and seeds `max_spawn_depth = 5` as the initial default (idempotent: existing values are never overwritten). ([#163](https://github.com/nova-openclaw/nova-cognition/issues/163))
- **`focus/agent-config-sync/README.md`** ‚Äî New comprehensive README documenting the plugin's dual-table sync, `agent_system_config` schema, supported config keys, configuration layering (installer default ‚Üí DB ‚Üí `$include` merge), and the full notification flow. - **Installer creates `notify_agent_config_changed()` DB trigger** ‚Äî `agent-install.sh` now automatically installs the `notify_agent_config_changed()` trigger function and attaches it to the `agents` table on fresh installs. - **`agent-config-sync` extension plugin** ‚Äî A gateway service that listens for PostgreSQL `LISTEN/NOTIFY` events on the `agent_config_changed` channel and writes `~/.openclaw/agents.json` with current agent model configurations. Uses atomic writes (temp + rename) and includes automatic reconnection with exponential backoff. Performs an initial sync on gateway startup. ([#146](https://github.com/nova-openclaw/nova-cognition/issues/146))
- **`$include` directive for `agents.json`** ‚Äî `openclaw.json` now includes `"$include": "./agents.json"` so that DB-synced agent configs are merged into the gateway config. - **Installer generates initial `agents.json`** ‚Äî `agent-install.sh` now queries the `agents` table at install time to produce a ready-to-use `agents.json`, so the config is correct even before the gateway starts. ([#146](https://github.com/nova-openclaw/nova-cognition/issues/146))
- **Installer sets `gateway.reload.mode`** ‚Äî The installer ensures hot-reload is enabled (required for the config sync to work); if the mode is unset or `"off"`, it is set to `"hot"`. ([#146](https://github.com/nova-openclaw/nova-cognition/issues/146))
- **`--no-restart` flag for `agent-install.sh`** ‚Äî the installer now auto-restarts the OpenClaw gateway (if running) after install so plugin changes take effect immediately; pass `--no-restart` to skip this and restart manually later ([#149](https://github.com/nova-openclaw/nova-cognition/issues/149))
- **Shared `pg` dependency install** ‚Äî `pg` is now installed to `~/.openclaw/node_modules/` (shared across extensions) instead of per-extension `node_modules/`.
